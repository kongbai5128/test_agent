**模块一：Context / Performance**

1.大模型面对第一轮长窗口或多模态输入时，first token 会显著变慢。有什么快速/低成本/用户体验也不差的方案？从5-10秒稳定压缩到2秒。

2.一个 session 连续聊了 200 轮，context 快爆了。你会怎么做压缩？如何确保压缩后的对话仍然流畅？

答：

短期记忆：

对200论对话进行压缩，采用三阶段：

1. 首先将最近的10轮保留，对早期190轮对话进行冗余信息的删除，包括ai的工具调用和tool返回结果等一个对话的中间过程都可以删除。
2. 如果仍然超阈值，对直接早期190轮对话内容进行压缩，不只是写“前文已省略”，而是生成结构化摘要，例如：用户目标、已确认决策、重要约束、当前进度、未完成问题、关键文件/工具结果、用户偏好、冲突处理规则。
3. 如果这样还是会超阈值，按照2的压缩规则，直接将过去的所有对话进行压缩。（还可以继续按照1和2的步骤继续压缩比如：3.最近的3轮保留，早期7轮对话进行冗余信息的删除；4.如果仍然超阈值，对早期7轮对话内容进行压缩。）

原始的完整对话仍然可以完整保留在磁盘，**需要追溯时再检索旧轮次**，比如用户问 第 50 轮的相关内容，不要靠摘要猜，直接从保存在磁盘的历史/RAG 取原文。

**长期记忆**

1. 把会话整合成长期记忆。这适合保存用户偏好、项目背景、长期约束，不会保存过多冗余内容。

最终形态是：

`system prompt + 相关长期记忆 + session summary + 最近 N 轮原文 + 当前用户消息`

**模块二：Memory**

1.和聊天 Agent 熟悉半个月后，用户问了一个以前问过的问题。Agent 如何做 memory 召回更合理？

2.你理解的 Agent memory 经典框架是什么？它的发展趋势是什么，最头部的玩家在怎么做？

答：

Agent Memory的经典框架是"写入-管理-读取（**Write-Manage-Read**）"闭环。写入时从原始对话提取成有价值的信息，管理中持续做合并、更新、去重来保持记忆库精简准确，读取时按需检索（语义/关键词/多步推理）

Agent memory 一般可以拆成五层：

1. **短期记忆 / 工作记忆**：当前 turn、最近 N 轮对话。
2. **情节记忆**：过去发生过什么，例如某次任务、某次用户反馈、某个已经完成的流程。。
3. **语义 / 事实记忆**：长期稳定事实，例如用户偏好、项目背景、常用技术栈、长期约束。
4. **程序 / 规则记忆**：Agent 以后应该怎么做，例如用户纠正过的工作方式、固定工作流。
5. **外部 / 组织记忆**：文档、代码库、知识库、业务数据库、MCP 工具等。

**发展趋势**

1. 从“长上下文暴力塞历史”走向“记忆提取 + 检索增强 + 小上下文注入”。因为长上下文有成本、延迟和注意力退化问题。
2. 从单纯 vector memory 走向 hybrid memory：向量 + BM25 + 实体关系 + 时间推理 + 权限过滤。
3. 从同步写入走向后台异步写入。memory extraction / consolidation 不应该影响用户首 token。
4. 从“自动记住一切”走向可控记忆：用户能开启/关闭、按 thread 控制、查看、删除，系统还要避免保存 secret。
5. 从个人记忆扩展到 `user / session / agent / app / org` 多租户作用域。不同用户、不同 Agent、不同团队的 memory 不能串。
6. 从文本记忆扩展到多模态和环境记忆，例如屏幕上下文、文件、浏览器、IDE、工具调用轨迹。
7. 从简单召回走向可评估 memory system：不仅看召回准确率，还看 token、延迟、成本、冲突处理和多 session / 时间问题能力。

**头部玩家怎么做**

Mem0主要通过“提取-更新-存储-检索”四个核心步骤来实现智能记忆，写入时用大模型从对话中提取事实并自动解决冲突与去重，存储时混合使用向量库、键值库和知识图谱；读取时通过语义搜索+元数据过滤+记忆衰减重排序，只把最相关最新鲜的记忆喂给模型。不够由于更新的操作关于复杂，它的新算法强调 ADD-only 抽取、去重、embedding、实体链接、时间元数据；召回时使用 semantic + BM25 + entity + temporal 的多信号融合。

Codex 的方向更偏工程 Agent：它把必须稳定执行的规则放在 `AGENTS.md`，把可复用工作流放在 Skills，把外部系统接入放在 MCP，自动 memory 只作为“有用上下文”的召回层。也就是说，重要规则不应该只存在 memory 里；memory 更适合记录稳定偏好、常用工作流、技术栈、项目坑点。Codex 还支持本地 memory 文件、后台提取/整合、thread 级别控制、secret redaction 等，这说明头部产品很重视可控性和安全边界。


**模块三：Task**

1.对于长程任务，大模型执行一段时间可能会忘掉目标，你知道哪些解决方案，有什么优缺？

2.用户给 Agent 下达任务：每天早上 9 点根据昨天聊天情况做复盘总结。你会怎么设计？

答：

这个需求我会设计成 **定时任务 + 会话检索 + 总结生成 + 结果投递**，不要依赖用户当天打开页面才触发。

用户创建任务时，先保存一条 schedule：

`user_id / timezone / cron=0 9 * * * / task_type=daily_review / enabled / last_run_at`

每天早上 9 点由 scheduler 触发 job。触发时先按用户时区计算“昨天”的时间范围，例如昨天 00:00 到 23:59，然后从 session store 里取这段时间的聊天记录、工具 trace、todo 变化、重要 memory。不要把所有原文直接塞给模型，先做过滤和压缩：

- 只取昨天有内容的 session。
- 用户消息和助手最终回复优先。
- 工具调用只保留关键结果和错误。
- 如果内容太长，先按 session 生成 digest，再汇总成一天的 digest。

然后调用模型生成复盘，输出结构可以固定：

```text
昨天主要聊了什么
完成了哪些事项
遗留问题 / 待办
重要决定
需要提醒用户的风险
今天建议关注什么
```

生成结果要作为一个 artifact 或 assistant message 持久化，带上 `job_id / date / source_session_ids / status`。这样用户之后可以追溯“这份复盘来自哪些会话”。

投递方式可以有两种：

- 如果用户在线，通过 SSE/WebSocket 推送“今日复盘已生成”。
- 如果用户不在线，写入 inbox / notification，用户下次打开时展示。

还要注意幂等和失败重试。每天同一个用户同一个日期只能成功生成一份复盘，`job_id` 可以用 `user_id + date + task_type`。如果模型调用失败，记录 `failed` 状态并重试；如果用户后来修改了昨天的会话，可以支持手动重新生成。


**模块四：Tool / Session Runtime**

1.Agent 工具有同步和异步两类。异步工具不能让用户一直等，但结果依然重要。你会如何设计异步工具执行和完成通知？

2.如果 session state 为 busy，此时用户又发来新消息，或者异步工具完成事件也到达，runtime 应该如何处理？

答：

核心原则：**同一个 session 只能有一个 Agent loop / worker 写状态**。`busy` 不代表丢弃新事件，也不应该直接开启第二个 loop，而是把用户消息和异步工具完成都当成事件，进入同一个 session event queue 串行处理。

session 状态可以简化成：

`idle / running / waiting_tool / cancelling / failed`

事件循环大致是：

```text
user_message
  -> agent_step
      -> assistant_message -> idle
      -> tool_start -> waiting_tool
            -> tool_done
                -> agent_continue
                    -> assistant_message -> idle
                    -> tool_start -> waiting_tool  # 继续循环
```

也就是说，`tool_done -> agent_continue` 之后不一定结束，模型可能继续调用下一个工具，也可能给最终回复。

如果 `running / waiting_tool` 时用户又发来新消息，普通消息先进入队列，前端显示“已加入队列”。等当前安全点结束后，再把这条消息作为新的 `user_message` 处理，或者在下一次 `agent_continue` 前合并进上下文。如果用户发的是“取消、停止、重新来”，则作为高优先级 interrupt，设置 `cancel_requested`，让当前 loop 在安全点停止。

如果 `busy` 时异步工具完成，也不要直接改 context 或直接调模型，而是写入 `tool_done` 事件，并带上 `session_id / run_id / tool_call_id / status / result`。worker 消费时先校验它是否属于当前任务版本；如果任务已经取消或结果过期，就记录 trace 但标记为 stale，不再注入模型。

最终要保证三件事：

- 同一 session 的状态写入串行化，避免 context 顺序错乱。
- 工具结果幂等处理，重复回调不会重复注入。
- 用户消息和异步工具事件都可追踪、可取消、可恢复。

**模块五：Agent Runtime 架构对比**

1.Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？他们各自这样设计的优缺点是什么？

答：

核心区别：**Claude / Claude Code 是 content block 模型，OpenAI-compatible function calling 是 `tool_calls + role=tool` 模型。**

Claude 的工具调用和工具结果都在 message 的 `content` 数组里：

```json
{
  "role": "assistant",
  "content": [
    {"type": "text", "text": "我先查一下"},
    {"type": "tool_use", "id": "toolu_1", "name": "read_file", "input": {"path": "app.py"}}
  ]
}
```

工具执行完后，结果也作为一个 content block 回给模型：

```json
{
  "role": "user",
  "content": [
    {"type": "tool_result", "tool_use_id": "toolu_1", "content": "文件内容..."}
  ]
}
```

GLM / 豆包等 OpenAI-compatible 接口更接近 OpenAI 的函数调用格式：模型在 assistant 消息的 `tool_calls` 字段里声明要调用什么工具。

```json
{
  "role": "assistant",
  "tool_calls": [
    {
      "id": "call_1",
      "type": "function",
      "function": {
        "name": "read_file",
        "arguments": "{\"path\":\"app.py\"}"
      }
    }
  ]
}
```

工具执行完后，应用追加一条 `role=tool` 消息：

```json
{
  "role": "tool",
  "tool_call_id": "call_1",
  "content": "文件内容..."
}
```

所以区别不是只有 `role=user` 和 `role=tool`，而是抽象方式不同：

Claude 是：

`content = text / image / tool_use / tool_result ...`

OpenAI-compatible 是：

`assistant.tool_calls -> app 执行工具 -> role=tool`

**Claude / Claude Code 这种方式的优点**

工具调用、工具结果、文本、多模态内容都在统一的 content block 里，表达能力更强。对于代码 Agent 来说，读文件、改文件、跑命令这些工具动作可以作为 runtime 事件被展示、审计和权限控制，所以更适合 Claude Code 这种工程 Agent。

缺点是协议更重，自己实现 runtime 时要处理 block 顺序、tool_use_id 对齐、权限、trace 展示等问题；而且和 OpenAI-compatible 生态不完全一样，适配不同模型成本更高。

**OpenAI-compatible function calling 的优点**

格式简单，核心就是：

`tools -> assistant.tool_calls -> role=tool -> 再调用模型`

GLM、豆包、DeepSeek、OpenAI 等模型都能按类似方式接入，业务系统实现起来更直接，适合查天气、查数据库、查订单、计算等普通业务工具。

缺点是它只规定“模型怎么表达我要调函数”，不负责 Agent Runtime。比如工具权限、执行日志、异步工具、session busy、上下文压缩、工具结果幂等，都要应用层自己设计。

一句话总结：

**Claude Code 更像“工具是 Agent 运行过程中的事件”；OpenAI-compatible function calling 更像“模型返回一段函数调用 JSON，后端自己执行”。**
